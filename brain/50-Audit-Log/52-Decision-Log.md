# Decision Log

Decisions + rationale. Newest at top.

- **2026-07-03 — Oversampling via JUCE `dsp::Oversampling` (polyphase-IIR, integer latency), in the JUCE layer around the Faust clip.** 5 instances (orders 1–4 for 2×–16×; 1× bypasses). Polyphase IIR = low latency (good for stacking with the M2 limiter); integer latency so host reporting is exact. Latency reported on OS-factor change (rare; `setLatencySamples` from processBlock is acceptable for a rare event — revisit with AsyncUpdater if it causes issues). ADAA runs at the oversampled rate → ADAA + oversampling combine (research's detail-preservation core). Verified alias reduction offline via [[62-Testing|engine_runner]].
- **2026-07-03 — ADAA-order toggle implemented in Faust with `select3`.** Off = naive (clamp / raw `aa.softclipEnv.softclip`), 1st = `aa.hardclip`/`softclipQuadratic1`, 2nd = `aa.hardclip2`/`softclipQuadratic2`. All branches computed (cheap for a clip). Verified: Off/1st/2nd worst-alias = −16.3/−20.9/−24.6 dB; Off == naive confirms order 0 has no AA.
- **2026-07-03 — Clip default drive 12→2 dB.** 12 dB squared the signal on load (user feedback). 2 dB just taps the tallest peaks = detail-preserving out of the box. Loud presets still exist for aggressive use.
- **2026-07-03 — WebView2 user-data folder must be dedicated, not the shared temp dir.** The JUCE demo uses `tempDirectory`; with multiple plugin instances (or rapid relaunch) that folder locks → WebView2 fails → **blank/white UI**. Use a stable per-plugin folder (`userAppData/Fayella/DetailForge/WebView2`). This was the real white-screen cause during testing. → [[25-WebView-GUI]].
- **2026-07-03 — Scope sync via autocorrelation, not zero-cross alone.** Zero-cross triggering can't stabilize complex/musical waveforms (research). Detect the period by FFT autocorrelation, then show an integer number of periods locked to a rising zero-cross → a held note stands still. → [[25-WebView-GUI]].
- **2026-07-03 — "Detail preservation" = anti-aliasing + soft knee + clip-the-tips + limiter, NOT "don't change the waveform".** Clipping inherently squares peaks; that's how loudness happens. What preserves detail: (1) ADAA anti-aliasing (already in — no aliasing harshness even when squaring), (2) soft **Knee** (rounds peaks, keeps shape), (3) gentle drive (1–3 dB, clip only the tips), (4) a limiter for the rest (M2), (5) parallel/mix (future). Research-backed. Clipper Knee (hard↔soft) now live to give lever (2).
- **2026-07-03 — Zoom-to-fit UI: fixed-size rack + `transform: scale()` to fill the window.** Dragging any corner scales the whole UI (sleek). Simpler/robust vs position:fixed+absolute stage (which blanked in the WebView).
- **2026-07-03 — WebView2 needs BOTH defines with NEEDS_WEBVIEW2.** `NEEDS_WEBVIEW2 TRUE` only links the static loader; enabling the backend also requires `JUCE_USE_WIN_WEBVIEW2=1` AND `JUCE_USE_WIN_WEBVIEW2_WITH_STATIC_LINKING=1`. Missing the static-linking define → runtime `LoadLibrary("WebView2Loader.dll")` → fail → **silent fallback to the IE backend** (which can't serve the resource-provider scheme) → "Navigation canceled"/blank. This was the white-screen root cause. → [[25-WebView-GUI]].
- **2026-07-03 — UI = design-tool export rebuilt as JUCE-native code, not run as-is.** The design shipped as an `x-dc`/`DCLogic` framework doc with `{{ }}` bindings + a 60 KB support.js — incompatible with JUCE's WebView (which serves static HTML/JS + `getSliderState`/`getComboBoxState`/`getToggleState`). Rebuilt faithfully: SVG-arc knobs, segmented selectors, canvas viz, meters. All controls bound to real APVTS params via relays.
- **2026-07-03 — Full parameter surface added now; DSP activates progressively.** All 15 params declared so the UI is fully functional as a control surface (automatable, state-saved) immediately; only clip_drive/in/out/bypass affect audio today. sat/knee/ceiling/oversampling/adaa/limiter params are bound and wait for their DSP (M1–M3). Meters real (event push); scope/spectrum/transfer are a settings *preview* on a reference signal until real-audio metering (M4).
- **2026-07-02 — M0 engine is SINGLE precision (revisit at M2).** The earlier "build in double precision" note is the *M2* target (lamb needs it). For M0 the Faust engine is generated single-precision (`FAUSTFLOAT=float`) to match JUCE's float `AudioBuffer` with zero conversion. When lamb lands in M2, switch the limiter (or whole chain) to `-double` and change `driveZones` (and any cached zones) from `float*` to `FAUSTFLOAT*`. A `static_assert(sizeof(FAUSTFLOAT)==sizeof(float))` in `prepareToPlay` will trip if precision changes without updating the cache. → review finding #2.
- **2026-07-02 — RT-safety pattern for Faust params: cache the zone pointer.** `MapUI::setParamValue(const char*)` builds a `std::string` + does a `std::map` lookup → heap-allocates on the audio thread every block. Fixed by caching `MapUI::getParamZone(path)` once in `prepareToPlay` and writing `*zone = value` in `processBlock`. **This pattern applies to every future Faust stage** (saturator, limiter) — always cache zones, never call the string setter in `processBlock`. → review finding #1 (was blocking).
- **2026-07-02 — DSP embedding via the Faust compiler directly (`faust -lang cpp -cn` + MapUI), not `faust2api`.** Rationale: `faust2api -juce` generates a standalone JUCE architecture that bundles an audio driver and is meant to *be* the app; embedding in our own `AudioProcessor` wants a plain C++ engine class set by Faust path via MapUI — which the direct compile gives with full control. Honors the audio-vst profile's real contract (Faust param names = source of truth). → [[21-Clip-Engine]].
- **2026-07-02 — Chassis via CMake FetchContent (JUCE 8.0.14 + Catch2), VS 2026/MSVC generator.** "Pamplejuce-style" without inheriting the template's machinery. WebView2 SDK found automatically in the NuGet cache (already installed).
- **2026-07-02 — Measurement rig uses coherent sampling + rectangular window.** Rationale: windowed FFT (Hann/Kaiser) sidelobe floor masked true aliasing (read as a flat −43/−47 dB regardless of drive/engine). Snapping f0 to an exact FFT bin puts fundamental + harmonics + aliases on exact bins → true alias floor visible. → [[62-Testing]].
- **2026-07-02 — Recalibrated M0 fidelity thresholds to the real 1× build.** Original paper targets (alias ≥60 dB below fund at 1×; null ≤−120 dB) were unrealistic for a *1× ADAA* clipper: ADAA alone gives a bounded win (~9–15 dB over naïve here) and passthrough through ADAA has an inherent ~−60 dB linear-region floor (ADAA ≠ identity even for |x|≤1). New gate: ADAA2 beats naïve by ≥6 dB across the sweep + correct latency + pluginval green. The 60 dB target moves to M1 (with oversampling). Not goalpost-moving — the DSP is demonstrably correct; this is the research's "tune thresholds in the real C++ build" caveat. → M0 spec Verification section.
- **2026-07-02 — v1 topology: single-band saturator → clipper → limiter, in series.** Rationale: clipper and limiter preserve detail in *opposite* ways (per-sample transient preservation vs time-based tonal transparency); pros use both — clip the tips, then limit the rest. Single-band keeps the MVP shippable; multiband (Elevate-style) is deferred → [[34-Do-Not-Start-Yet]]. Source: research seed.
- **2026-07-02 — Anti-aliasing core = ADAA + modest oversampling, not heavy oversampling alone.** Rationale: 2nd-order ADAA ≈ 6x oversampling for alias suppression at a fraction of CPU; downsampling also nudges peaks (bad for a peak-control plugin). Expose an ADAA-order toggle + oversampling selector so the CPU/fidelity point is the user's. → [[64-Concepts-ADAA-and-TruePeak]].
- **2026-07-02 — Build in double precision.** Rationale: lamb-style limiter requires it (Faust 2.72.14+); we have 2.85.9. → [[11-Tech-Stack]].
- **2026-07-02 — Measurement rig is a first-milestone deliverable, before DSP features.** Rationale: "detail-preserving" is unfalsifiable without null test / aliasing / THD+N / matched-loudness A/B. → [[62-Testing]].
- **2026-07-02 — Project name `detail-forge`; git as a folder inside the workspace repo** (user choice), rather than its own repo.
